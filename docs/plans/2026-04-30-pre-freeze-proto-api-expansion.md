# Pre-Freeze Proto API Expansion Implementation Plan

**Goal:** Execute the final pre-freeze breaking protobuf hard cut so `arco.*.v1` becomes the durable public API surface for the expanded catalog, metastore, governance, and control-plane contracts.

**Architecture:** Keep the package names at `arco.*.v1` because this is the explicit pre-freeze stabilization cut, then make broad breaking changes in one controlled slice. Add a dedicated metastore/governance proto instead of inflating `catalog.proto`, wire it into `arco-proto` generation, update request validation and adapters, then regenerate the frozen post-hard-cut baseline. After this plan lands, proto changes must be additive unless a future `v2` package is introduced.

**Tech Stack:** Protobuf, Buf, `prost`/`tonic`, Rust, `serde`, `cargo test`, `cargo xtask proto-breaking-check`

## Scope

- Keep `arco.common.v1`, `arco.catalog.v1`, `arco.controlplane.v1`, and `arco.orchestration.v1` as the durable public packages after this cut.
- Add `proto/arco/catalog/v1/metastore.proto` for broader metastore/governance object and mutation contracts.
- Model stable object IDs explicitly for grants, credentials, external locations, bindings, governance attachments, volumes, functions, registered models, and model versions.
- Expand control-plane transaction mutation envelopes so the new metastore operations can be committed through the same authoritative path as catalog DDL.
- Update generated Rust module inclusion, compatibility re-exports, validation helpers, request hashing, adapters, and golden fixtures.
- Regenerate `proto-baselines/post-hard-cut-v1.binpb` only after the expanded API is intentional and tests pass.

## Non-Goals

- Do not implement the full catalog compactor/state projections in this plan. That belongs to the authoritative metastore governance plan.
- Do not add a `v2` package unless review decides this is no longer a pre-freeze cut.
- Do not use opaque `properties` or JSON blobs as the primary governance contract.
- Do not expose system tables here except where tests need names reserved for later projection work.

## Execution Notes

- Use `@test-driven-development` for code tasks.
- Use `@receiving-code-review` if proto/API feedback challenges field names, identity semantics, or package placement.

## Acceptance Criteria

- The repo clearly documents that this is the final pre-freeze hard cut for `arco.*.v1`.
- The expanded metastore/governance contract is generated by `arco-proto` and covered by contract tests.
- Stable IDs are first-class fields in the new metastore/governance messages.
- Control-plane request validation rejects empty metastore mutations the same way it rejects empty catalog and orchestration mutations.
- Existing catalog/orchestration transaction behavior still passes after adapter updates.
- `proto-baselines/post-hard-cut-v1.binpb` is regenerated after the new API shape is complete.
- CI enforces the regenerated baseline through `cargo xtask proto-breaking-check`.
